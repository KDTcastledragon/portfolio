대규모 메시지 처리 지향
웹 채팅, CastleChat

메시지 유실 없는 Kafka 전송 파이프라인과 Redis 실시간 읽음 동기화를 5개 프로세스 아키텍처로 직접 설계하고 구현했습니다.

배포 링크

설계 핵심

01
5개 프로세스 분리

도메인 · 게이트웨이 · 엔진 · 워커 · AI를 독립 프로세스로 격리해 장애와 배포 영향 범위를 좁혔습니다.

02
Kafka durable 파이프라인

전송 응답 전에 durable ACK를 보장하고, DB 쓰기는 전용 워커로 분리했습니다.

03
Redis write-behind 읽음 동기화

읽음 위치는 Redis에 원자적으로 병합하고 주기 배치로 DB에 반영합니다.

CastleChat Logical Architecture

CURRENT CODE + NGINX TARGET

Nginx Docker reverse proxy를 기준으로 정리한 요청·이벤트·저장 흐름입니다.

요청·응답 Kafka 이벤트 공유 Redis
Frontend React Client HTTP · WebSocket
Docker Nginx Reverse Proxy route · TLS termination
Process 01 Domain Service REST · upload · snapshot query
Process 02 WebSocket Gateway session · routing · broadcast
Process 05 AI Assistant Service context query · reply suggestion
Process 03 Channel Engine validation · orchestration · publish
Event Log Kafka durable ACK · retry · DLT
Process 04 Event Persist Worker idempotent message DB write
Shared State Redis Spring Session · room member · LRM · dirty read position
External API Gemini LLM message recommendation
Primary DB MariaDB domain · room · message persistence
  1. Entry

    React Client ↔ Nginx Reverse Proxy

  2. REST Domain Flow

    Nginx → Domain Service ↔ MariaDB
    Domain Service ↔ Redis Shared Session

  3. Realtime Message Flow

    Nginx ↔ WebSocket Gateway ↔ gRPC ↔ Channel Engine
    Channel Engine → Kafka → Event Persist Worker → MariaDB

  4. Realtime State

    Channel Engine ↔ Redis
    Channel Engine ↔ MariaDB

  5. AI Assist Flow

    Nginx → AI Assistant Service ↔ MariaDB Read Only
    AI Assistant Service → Gemini API

Kafka event pipeline. SEND·DELETE·REACT는 상태 변화 보존과 방 단위 순서가 중요해 durable ACK 이후 응답하고 DB 쓰기는 별도 프로세스로 격리합니다.
Redis scheduled write-behind. READ는 중간 이벤트보다 최신 LRM이 중요해 원자적으로 최댓값만 병합하고 dirty 위치를 주기적으로 DB에 반영합니다.
